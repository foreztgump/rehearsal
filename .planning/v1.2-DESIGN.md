# v1.2 — Polish & Hardware-Aware Pipeline — Design

## Intent

v1.2 turns Adept from a single-GPU (RTX-5090-class) tool into a **hardware-aware** one that runs
on varied consumer machines — down to ~8GB-VRAM cards and AMD — and is **idiot-proof to install** —
while also fixing the STT accuracy/cut-off escalated out of Phase 15 and polishing conversation
feel and the avatar. One milestone, six active streams (UI/UX polish already shipped in Phase 13/14 —
see the R4 note), sequenced so each foundation lands before what depends on it.

This is a **release-level design**: scope, sequencing, key decisions, success criteria, and risks.
**Each stream gets its own detailed design + plan** when its build cycle starts. The local-first,
latency-first, and persona-as-sole-guardrail invariants are unchanged.

## The organizing concept: three hardware tiers

Instead of one fixed stack, v1.2 defines the stack **per detected hardware tier**. Every model and
engine decision answers "which tier am I on." R2 *defines* the Floor, R3 *selects engines/models*
per tier, R6 *adds AMD* as another detected axis, R7's installer *detects → confirms → writes the
profile*.

| Tier | Target | LLM | STT (default) | STT (opt-in accuracy) | TTS |
|---|---|---|---|---|---|
| **Floor** | ~6GB whole-stack / 8GB cards (or CPU-only) | abliterated small Qwen3 (≈1.4–3.1GB q4) | streaming Nemotron **CPU-ONNX** | **Parakeet-tdt-0.6b-v2 offline (CPU)** — higher latency accepted | Kokoro-GPU (~1GB) or CPU-ONNX |
| **Capable** | ~12GB | E2B / E4B (today's class) | streaming Nemotron `[70,6]` GPU (~1.7GB) | Parakeet-offline (CPU or GPU) | Kokoro-GPU |
| **Best** | 24GB (RTX 5090) | E4B (+ future "Best" model, v1.3) | streaming Nemotron `[70,13]` GPU | Parakeet-offline GPU (WER 6.05%) | Kokoro-GPU (+ Chatterbox v1.3) |

**Streaming is the latency-safe default on every tier; the buffered Parakeet accuracy engine is a
selectable opt-in on every tier**, including Floor/CPU-only where higher latency is explicitly
acceptable in exchange for accuracy.

## Spine & sequencing

**R2 → R3 → R6 → R7, then R1 ∥ R5.**  (R4 UI/UX is already delivered — not a v1.2 stream.)

- **R2 (keystone)** — model lifecycle + the ~6GB Floor profile. Defines the budget every tier fits
  inside; gates everything below. **Supersedes** the v1.1 "16GB / two-resident-LLM / `keep_alive=-1`"
  assumption.
- **R3** — hardware-aware engine/model selection: the per-tier STT engines (streaming default +
  buffered Parakeet opt-in on all tiers), the `finalize()` tail-drop investigation, and generalizing
  the selectable-engine seam to TTS (Chatterbox-ready, not built). Depends on R2's budget.
- **R6** — AMD/ROCm as a detected hardware axis feeding tier selection. Depends on R2/R3 engine defs.
- **R7** — the one-line installer that detects hardware, confirms, and writes the tier profile that
  R2/R3/R6 defined. **Last by construction** — it installs what the others define.
- **R1 / R5** — polish (conversation feel, avatar). Frontend/tuning, don't gate each
  other, so they run in parallel at the tail against the folded latency baseline. (R4 UI/UX already
  delivered — see note.)

## Entry / exit criteria

- **Entry:** none blocking. v1.1 is feature-complete; its unsigned operator gates **fold into v1.2's
  single verification pass** (user decision). The obsolete v1.1 16GB co-residency matrix is dropped —
  superseded by R2's ~6GB tier measurements.
- **Exit (single verification pass):**
  - 3-tier VRAM matrix measured at the KB-load prefill peak (q8_0 engaged where applicable).
  - **P50 < 1.0s held on Capable + Best tiers** (both LLM choices, new STT leg).
  - **Floor tier latency *characterized*** and stated honestly — a small model + CPU STT/TTS won't
    always beat 1.0s; that is an honest tier contract, not a failure. Buffered Floor STT is expected
    to exceed 1.0s and is documented as such.
  - **AMD tier: functional, best-effort — not P50-guaranteed** (see R6 risk).
  - Folded v1.1 gates: PERF-04 (both LLMs) + re-scoped Phase 9/10/11 GPU gates + Phase-14 B1–B4.
  - Per-engine `verify-build` gate (incl. abliterated floor-LLM red-team).

## Stream designs

### R2 — Model lifecycle & the ~6GB Floor (keystone)

**Lifecycle rework:** single-resident LLM via Ollama's native **`OLLAMA_MAX_LOADED_MODELS=1`**
(loading the picked model evicts the prior one — never two LLMs resident, the 6GB-floor invariant).
`keep_alive` **stays `-1` (always-resident) by default** so the resident model isn't cold-reloaded
after idle — that protects the P50<1.0s metric, and `MAX_LOADED_MODELS=1` already caps residency at
one, so a finite idle-unload would only add latency for no VRAM gain. Both are env-tunable
(`OLLAMA_MAX_LOADED_MODELS=2` on a roomy GPU = warm Fast↔Better swaps; finite `OLLAMA_KEEP_ALIVE`
frees idle VRAM on a shared host). *(Revised from "drop keep_alive=-1" per the R2 pre-merge code
review — single-residency is delivered by MAX_LOADED, not by dropping the pin.)*

**The Floor budget is comfortable** (8GB card → ~6GB usable after headroom):

| Floor variant | Kokoro | STT | LLM | GPU total |
|---|---|---|---|---|
| **B (likely default)** | ~1GB GPU | CPU-ONNX (0 GPU) | abliterated Qwen3-4B-class ~3.1GB | **~4.1GB** |
| A (weak-CPU variant) | ~1GB GPU | Nemotron streaming ~1.7GB | abliterated Qwen3-1.7B `/no_think` ~1.9GB | ~4.6GB |

Both fit ~6GB with headroom, so the Floor is not a knife-edge. Final default chosen by measurement
(CPU-contention check — see Risks).

**Kokoro "5GB" is already solved.** The model is 82M params (~0.33GB); the rest is PyTorch CUDA
context + caching-allocator *reserved* memory. This box already measures **~954 MiB warmed** with
`expandable_segments` (`STACK.md:183`), down from ~1832 MiB. The only cleanup is the **stale
`docker-compose.yml:224` "~5GB observed" comment**. CPU-ONNX Kokoro stays an *optional* lever (frees
the GPU entirely but stacks CPU load — 15a already lists "no CPU-Kokoro" as a non-goal), not required.

**Floor LLM** — abliterated small model so the persona stays the sole guardrail consistently across
all tiers (user decision, matching the Phase-8 abliterated stance). Candidate verdicts:

| Candidate | Verdict |
|---|---|
| `lfm2.5-thinking:1.2b` | ❌ thinking-by-design (violates thinking-OFF) — use the `lfm2.5-1.2b-instruct` sibling if LFM2 is wanted |
| `minicpm-v4.6:1b` | ❌ MiniCPM-**V** = vision/multimodal; vision tower is dead weight for a voice app |
| `smollm2:1.7b` | ⚠️ real but weaker (late-2024) — fallback only |
| `roshansoni/qwen-3.5-2b-q4_k_m` | ⚠️ use first-party `qwen3.5:2b` instead of the single-uploader community tag |

R2's build cycle sources a **trustworthy abliterated small Qwen3 GGUF** (Qwen3-4B-Instruct-class
preferred; 1.7B `/no_think` fallback for the STT-on-GPU variant) and runs it through the existing
Phase-8 `verify-build.sh` gate (chat-template sanity + thinking-off artifact scan + persona red-team).
Note: Gemma 4's smallest variant (E2B ≈ 7.2GB) exceeds the Floor budget — it *is* the Capable tier.

### R3 — Hardware-aware engines & models

**STT — three tiers, real numbers** (streaming default + buffered Parakeet opt-in on *every* tier):

| Tier | Default (streaming) | WER | VRAM | Opt-in accuracy (buffered) |
|---|---|---|---|---|
| Floor | Nemotron int8/int4 **ONNX (CPU)** | ~7–8% | 0 GPU (~0.7–0.9GB RAM) | **Parakeet-tdt-0.6b-v2 offline (CPU)** — higher latency OK on low-end HW |
| Capable | nemotron-streaming `[70,6]` | 7.07% | ~1.7GB | Parakeet-offline (CPU or GPU) |
| Best | nemotron-streaming `[70,13]` | 6.93% | ~1.7GB | Parakeet-offline GPU (WER 6.05%) |

- **Streaming stays the default everywhere** — its finalize cost is *flat* (~100ms, independent of
  utterance length); buffered finalize *grows with length* and only fits the post-EOU budget for ~5s
  turns on strong GPUs. The latency math is the reason streaming is non-negotiable for the default.
- **15b revised:** `parakeet-tdt-0.6b-v2` is **offline-only** (forcing it to stream nearly doubles
  WER) → it is the **buffered accuracy engine** and **replaces faster-whisper** as the first one
  built: lower WER (6.05% vs ~7.4%), cheaper TDT decoder, **NeMo-native (no CTranslate2 dep
  conflict)**. faster-whisper drops to "documented alternative, not built."
- **`finalize()` tail-drop — honest scope:** the canonical NeMo fix (`previous_pred_out` threading)
  was **already tried in 15a and regressed** (`keep_all_outputs` is already set every chunk). R3
  **budgets real GPU investigation** (trailing-silence pad done correctly, or a deeper root cause) —
  *not* a promised one-liner. The buffered tiers sidestep the bug entirely (no lookahead → no held
  tail).
- **Selectable-engine seam generalizes to TTS:** add a `TTS_ENGINE` env + compose profile mirroring
  the existing `STT_ENGINE` pattern, via the existing `KOKORO_BASE_URL` indirection — **Chatterbox-
  ready for v1.3, not built now** (YAGNI). Caveat: the avatar's Path-B word-accurate lip-sync reads
  Kokoro's bespoke `/dev/captioned_speech` word timestamps, so non-Kokoro engines degrade to Path-A
  (the already-documented per-utterance fallback). Voice path is unaffected (standard OpenAI contract).
- **Selection is install-time, not a live in-app switch:** `gpu-doctor.sh` detects VRAM/vendor →
  writes the tier profile; builds on the existing `placement.py` / `STT_RUNTIME` / `STT_ENGINE` seam.

### R6 — AMD / ROCm

**Minimal AMD profile (v1.2):** `Ollama-ROCm LLM on GPU` + `CPU-ONNX STT` + `Kokoro CPU-ONNX or
PyTorch-ROCm`. AMD ≈ "CPU-heavy profile + ROCm LLM."

- **LLM → Ollama-ROCm (the solid GPU win):** official `ollama/ollama:rocm` image + `--device
  /dev/kfd --device /dev/dri`. RDNA3 (gfx1100, RX 7900) most reliable; RDNA2 (gfx1030, RX 6800/6900)
  supported on Linux; lesser cards need `HSA_OVERRIDE_GFX_VERSION`. Low effort (image + env swap).
- **STT → CPU-ONNX (forced default on AMD):** NeMo-GPU on ROCm is unverified — its fast RNN-T decoder
  uses CUDA Graphs (won't run on HIP) and the cache-aware streaming path is unproven on AMD. CPU-ONNX
  is AMD-agnostic and already the Floor path. (NeMo-on-ROCm is *not impossible* — a community project
  runs NeMo Parakeet on ROCm — so it's a deferred stretch, not a dead end.)
- **TTS → Kokoro CPU-ONNX (zero-risk) or PyTorch-ROCm** on a strong RDNA3 card (needs gfx override +
  MIOpen tuning).
- **Honest contract:** voice-to-voice **P50 < 1.0s on AMD is entirely unmeasured and may not hold**
  with STT on CPU. AMD ships **functional / best-effort, not P50-guaranteed** in v1.2. **Linux-only**
  (Windows RDNA2 is unsupported by ROCm).
- **Deferred past v1.2:** NeMo-GPU-STT on ROCm; GPU-ONNX via MIGraphX (the onnxruntime ROCm EP was
  removed in 1.23) — both genuine engineering needing real AMD hardware.

### R7 — Idiot-proof installer

One-line curl-to-shell that:
1. **Detects hardware** — extends `gpu-doctor.sh`: vendor (NVIDIA / AMD / none) + VRAM.
2. **Maps to a tier profile** — Floor / Capable / Best (+ AMD variant).
3. **Confirms with the user** before changing anything.
4. **Writes `.env`** with the resolved profile (model tags, `STT_ENGINE`/`STT_RUNTIME`,
   `TTS_ENGINE`, compose profiles).
5. **Pulls/builds the right images + models** and starts the stack.
6. **Start/stop docs.** Builds on the existing `install.sh` / `up.sh` / `down.sh`.

### R1 — Conversation feel (polish)

Tune barge-in reliability, EOU/VAD margins, and perceived latency against the folded v1.1 latency
baseline. Includes backlog #8 (VAD/interrupt margin A/B: 0.65→0.60 / 0.30→0.25 for speaker echo;
revert if it misfires). Distinct from R3's STT cut-off (that is a model/finalize bug, not turn-taking).

### R4 — UI/UX polish — ALREADY DELIVERED (not in v1.2 scope)

The UI/UX overhaul this stream covered — the `design-mockups/v4` direction + landing/setup-before-
connect + talking-screen polish + clean disconnect/reconnect — shipped in **Phase 13** (+ the Phase-14
End→reconnect fix). Retained here only for R-number traceability against the ROADMAP awareness map.
If any sub-item is still wanted, pull it into R1 or a small follow-up — it is not a v1.2 stream.

### R5 — Avatar polish (polish)

- **Emotion/expression** (`setMood` / morphs driven by conversation state or sentiment) + idle and
  speaking **body motion** — once expressiveness exists, full-body framing becomes a viable default
  (backlog #4).
- **Word-accurate lip-sync:** confirm Path-B fires when the avatar is ON; **fix the multi-sentence
  anchoring bug** — today only the first sentence anchors (a schedule is popped only on a
  silence→sound edge), so later sentences fall back to rough Path-A (backlog #3).
- **Verify captioned-TTS first-audio doesn't regress** P50 avatar ON vs OFF; stream the Kokoro
  `/dev/captioned_speech` response incrementally if it does (backlog #5).

## Out of scope → v1.3+

- **Chatterbox TTS** (Resemble AI, MIT; Turbo 350M ~6–8GB) — seam designed in R3, **built in v1.3**
  (user's call). Sized to the Best/Capable tier.
- **NeMo-GPU-STT on ROCm** + **GPU-ONNX via MIGraphX** — real-AMD-hardware engineering.
- **faster-whisper engine** — documented alternative only; Parakeet-offline is preferred.
- **Live in-app engine switching** — engine choice is install-time (YAGNI).
- LLM-F1 (per-persona model defaults), LLM-F2 (24GB "Best" model — Gemma-26B / Qwen3-8B),
  AVTR-F1/F2, STT-F1 (cyber-vocab fine-tune), multilingual STT.
- **Watch-list (not now):** Qwen3-TTS, CosyVoice 3.0 (newer Apache-2.0 streaming TTS), Moonshine v2
  (edge/CPU streaming STT for a future Floor option).

## Carried risks & unknowns (resolved in build cycles, need measurement)

1. **STT cut-off fix is not guaranteed** — the canonical NeMo fix regressed in 15a; R3 budgets GPU
   investigation, with the buffered tiers as the accuracy escape hatch if the streaming fix stays
   elusive.
2. **Batch-1, short-utterance buffered finalize latency on consumer GPUs is unmeasured** — decides
   exactly where buffered Parakeet fits on the hot path vs strictly off it.
3. **AMD P50<1.0s is unmeasured** — the real product risk behind the best-effort AMD contract.
4. **Floor CPU contention** — STT-CPU-ONNX + LLM (+ optional TTS-CPU) on shared cores; measure P50
   before committing Floor variant B as default.
5. **Abliterated small-model sourcing/trust** — single-uploader community tags; mitigated by the
   verify gate + a trusted source.
6. **Parakeet-offline VRAM grows with utterance length** (~2–5GB batch-1) — prefer CPU/RAM on the
   Floor to protect the GPU budget; cap/verify for 30s+ interview turns.

## R2 — deferred code-review findings (route to the owning round)

The R2 pre-merge code review (high-effort, 28 agents) confirmed several **floor-path** issues that
are dormant in the shipped default config (`fast`/`better`, `ADEPT_DEFAULT_MODEL` unset) and belong
to the round that activates the floor tier. Tracked here so they aren't lost:
- **R7 (installer/web):** `ADEPT_DEFAULT_MODEL=floor` with an unset `OLLAMA_MODEL_FLOOR` makes
  `build_session` SystemExit at startup (the model.update RPC path is guarded; the startup path is
  not) — R7 must only set the floor default once the floor tag is pinned, or add a graceful fall-back
  to `fast`. The web `ModelPanel` also can't select `floor` and would mis-send `fast` on a floor host
  (OOM risk); and prewarm warms the Fast tag, not the env-driven default — R7 threads the chosen
  default through warmup + the picker.
- **Task 5:** `FLOOR_LADDER` is now wired to **abliterated Qwen3-4B-Instruct-2507** (non-thinking by
  construction — rung 1) → **abliterated Qwen3-1.7B** (rung 2) → first-party `qwen3.5:2b-q4_K_M`
  (non-abliterated last resort, rung 3). Still pending `verify-build.sh` + the VRAM measure at the
  R2 GPU test before it's considered verified.
- **R3 (hardware-aware placement):** `placement.py` `VRAM_TOTAL_MB` is hardwired to 16384 — the floor
  tier needs a hardware-derived total before its `STT_HEADROOM_MEASURED` GPU branch is safe on 6–8GB.
- **Noted (non-blocking):** `MODEL_CHOICES`/`_MODEL_ENV` could derive one from the other; the
  `session.llm._opts.model` swap is a pre-existing Law-of-Demeter reach into a LiveKit internal.

## Per-stream success criteria

- **R2:** one LLM resident at a time (`OLLAMA_MAX_LOADED_MODELS=1`); a Floor profile boots and holds
  a conversation in ~6GB on an 8GB card; the Fast/Better switch swaps (no two-resident); `keep_alive`
  default `-1` retained (P50-safe; env-tunable); stale Kokoro compose comment fixed.
- **R3:** `STT_ENGINE` selects streaming Nemotron (default) or buffered Parakeet-offline on any tier;
  per-tier defaults resolve at install; the `TTS_ENGINE` seam exists (Kokoro only shipped); the
  tail-drop is either fixed (no-WER-regression check) or documented as mitigated with the buffered
  escape hatch.
- **R6:** the stack comes up on a supported AMD card with the LLM on ROCm + STT on CPU-ONNX; the
  AMD latency contract is documented honestly.
- **R7:** a fresh machine goes from one command → detected tier → confirmed → running stack, on both
  NVIDIA and AMD, with start/stop docs.
- **R1:** barge-in/EOU/perceived-latency tuned against a measured baseline with no P50 regression.
- **R5:** avatar shows emotion + body motion; Path-B anchors every sentence (not just the first);
  captioned-TTS first-audio shows no P50 regression.
