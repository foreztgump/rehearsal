# v1.2 — R3 — Hardware-aware engines & models — Design

## Summary

R3 closes on **buffered non-streaming Parakeet** as the supported STT path. On good
NVIDIA hardware, `nemo-stt` runs `nvidia/parakeet-tdt-0.6b-v2` through NeMo
`ASRModel.transcribe()` on GPU; the CPU service keeps a sherpa-onnx Parakeet fallback.
The old Nemotron `streaming` and `hybrid` modes remain in code as manual legacy
comparison modes, but they are no longer recommended defaults.

This supersedes the earlier R3 design assumption that GPU tiers should default to
`hybrid`. Live smoke testing found GPU `buffered` accurate enough and natural enough
at `STT_STREAM_CHUNK_MS=320` + `STT_ENDPOINT_SILENCE_MS=640`, while the streaming
path continued to show cut-off/accuracy problems.

**Why R3 exists:** 15a UAT found streaming 0.6B STT systematically drops trailing words at
end-of-speech (the lookahead holds a tail the hard cutoff never completes); the surgical fixes
regressed. A **buffered** engine has no lookahead, so the cut-off is structurally absent. That — plus
silence-robustness — is the win, **not** a benchmark-WER leap (see D18).

## Engine model — `STT_ENGINE` is a 3-mode selector

`STT_ENGINE ∈ {streaming, buffered, hybrid}` is the **mode**; the existing `STT_RUNTIME ∈ {gpu, cpu}`
stays the **device** for the streaming Nemotron engine. Orthogonal axes.

| `STT_ENGINE` | Runs | Live partials | Authoritative final (→ LLM) | Endpointing | P50<1.0s hot path? |
|---|---|---|---|---|---|
| `buffered` | Parakeet only | none | Parakeet transcribe at EOU | **VAD/RMS energy** | selected path |
| `streaming` | Nemotron only | yes (Nemotron) | Nemotron `finalize` | Nemotron + VAD | legacy comparison |
| `hybrid` | Nemotron **+** Parakeet | yes (Nemotron, display-only) | Parakeet correction | Nemotron + VAD | legacy experiment |

The finalized path uses `buffered`; there are no live partials. The message bubble is
updated from the authoritative Parakeet final once the turn finalizes.

## Per-tier defaults (resolved at install, advise-only)

| Tier | Hardware | Default | Opt-in | Parakeet device |
|---|---|---|---|---|
| **Floor** | ~6–8GB / CPU-only | `buffered` | legacy `streaming` only for comparison | CPU sherpa-onnx Parakeet |
| **Capable** | ~12–16GB | `buffered` | GPU only if placement proves fit | CPU sherpa-onnx Parakeet |
| **Best** | 16GB+ NVIDIA | `buffered` | legacy `streaming`/`hybrid` only for comparison | GPU NeMo Parakeet |

Good NVIDIA hardware should run `docker compose --profile stt-gpu up` with
`STT_FORCE_CPU=0` and `STT_HEADROOM_MEASURED=1` so `buffered` uses the GPU Parakeet
backend for speed.

## Architecture — server-side composition over the existing seam

**Chosen: server-side composition.** Rejected: *agent-side* (two STT plugins merged in the agent —
reaches into the LiveKit turn pipeline + endpoint-authority invariant, riskier) and *single
mega-backend* (hides three behaviors in one module, kills the per-module seam's testability).

The server (`stt/server.py`) gains a **`primary` backend** (drives deltas + EOU) and an optional
**`final` backend** (authoritative transcript), both via the existing `importlib`-by-name dispatch.
Pure composition over today's 5-callable seam (`load_model / new_stream_state / decode_chunk /
finalize / reset_turn_state`):

| `STT_ENGINE` | primary (deltas + EOU) | final (authoritative) |
|---|---|---|
| `streaming` | `backend_nemo` / `backend_onnx` (Nemotron) | same — Nemotron `finalize` |
| `buffered` | `backend_parakeet_nemo` on GPU / `backend_parakeet` on CPU (accumulate PCM, **no deltas**, energy-EOU) | same backend (transcribe buffer) |
| `hybrid` | `backend_nemo` (Nemotron deltas + EOU, **unchanged**) | `backend_parakeet` (transcribe accumulated PCM at EOU) |

### Components
- **`stt/backend_parakeet.py` (new)** — implements the same 5 callables, but `decode_chunk` *appends
  PCM to `state["_turn_pcm"]`* and returns no growth; `finalize` runs **Parakeet-tdt-0.6b-v2 int8
  ONNX** over the whole buffer once (encoder ONNX + a TDT greedy-decode loop in numpy — the **same
  ORT + numpy-decode pattern `backend_onnx` already uses** for the Nemotron CPU port). A
  module-level **`STREAMS = False`** capability flag is the only new seam concept.
- **Server EOU policy keys on `STREAMS`** (today's logic, generalized): `STREAMS=True` → emit deltas
  + text-stall EOU (`_emit_delta`, unchanged). `STREAMS=False` → no deltas + a **cheap RMS
  energy-silence EOU** (numpy mean-square per chunk, reusing `ENDPOINT_SILENCE_MS`) — there is no
  growing transcript to stall-detect, so silence is detected acoustically. This is the *one* new
  piece of server logic. `hybrid` needs none (Nemotron still drives EOU).
- **`STT_ENGINE` gates which `load_model()` fires at lifespan startup** → `streaming` loads only
  Nemotron (zero Parakeet RAM/VRAM even though the image carries it). No wasted memory, disk-cost only.
- **Device for Parakeet** comes from `resolve_parakeet_device(env)` (see Hardware awareness) →
  selects the onnxruntime EP (`CPUExecutionProvider` vs `CUDAExecutionProvider`).

### Agent-plugin impact: minimal
The WS contract (`ready` / `delta` / `final`) is byte-identical. `buffered` simply omits `delta`s;
`hybrid` sources `final` from Parakeet. The plugin (`agent/nemo_stt.py`) already **commits a turn on
FINAL alone** (`_audio_transcript` is filled only by finals; interims are "still-speaking" signals),
so a zero-interim turn commits cleanly. Two small touches: a per-engine **metrics label** (so `stt_ms`
attributes correctly) and wiring the **autonomous-final latency span** server-side (the buffered/hybrid
EOU final has no flush sentinel, so today's `duration` would read 0.0 — the G3 gate needs the real
EOU→final span).

### Image / compose isolation — no new image
Because the buffered engine is **lean ONNX-int8 (no torch, no CTranslate2)**, both existing STT images
just gain the `backend_parakeet` module + its int8 ONNX weights, gated by `STT_ENGINE`:
- **`stt/Dockerfile` (GPU)** — already NeMo/torch (Nemotron). Add `onnxruntime-gpu` + Parakeet ONNX
  weights → serves `streaming` + `hybrid` (Parakeet on the CUDA EP). One process.
- **`stt/Dockerfile.cpu` (lean ORT)** — already onnxruntime CPU (Nemotron ONNX). Add Parakeet ONNX
  weights → serves Floor `streaming` (Nemotron) **and** `buffered` (Parakeet) from one image; the mode
  picks which `load_model` fires. No torch added.
- Compose **profiles** stay as today's `stt-gpu`-vs-default split; `STT_ENGINE` selects the mode within
  the chosen image. (Parakeet ONNX weights are ~0.6–0.7GB on disk, loaded only when the mode needs them.)

### Error boundaries
- Parakeet model/weights load failure → `SystemExit` at startup (mirrors the `STT_MODEL` KeyError
  posture) → `/health` stays 503 → the agent's `service_healthy` gate never starts on a broken image.
- Transcribe exception → emit `{"type":"error"}` **and** a VAD-driven empty `final` so the turn never
  deadlocks (the framework waits for a FINAL).
- A **max-buffer cap** (`STT_BUFFERED_MAX_S`, default 60s) bounds `state["_turn_pcm"]` so a missed EOU
  can't grow memory unbounded; on cap, force-finalize.
- Empty/garbled Parakeet output → empty FINAL → turn still commits (VAD-driven), LLM sees empty (same
  contract as today's empty-STT boundary).

## Hardware awareness

Three coordinated changes; `placement.py` stays pure (never probes hardware, never raises,
sandbox-runnable via `_self_check`) and `gpu-doctor.sh` stays advise-only (never mutates `.env`,
always exits 0).

### A. `placement.py` — env-derived total + a Parakeet fit gate
- `VRAM_TOTAL_MB` becomes an **env read** threaded through the existing `env` Mapping (default `16384`
  when unset) — *not* an in-module `nvidia-smi` probe (that would break purity). The real total is
  supplied by gpu-doctor. The `STT_HEADROOM_MEASURED` gate stays the safety net: even if the total
  defaults high on an 8GB card, the GPU branch stays CPU until an operator measures and flips the flag.
- `resolve_stt_placement(env)` chooses GPU iff
  `max(LLM_PEAK) + KOKORO_MB + STT_GPU_MB <= ceiling`; otherwise it stays on CPU. Same
  `STT_FORCE_CPU` / `STT_HEADROOM_MEASURED` gating posture as Phase 10.
- `STT_GPU_MB` is now the rounded-up observed GPU Parakeet resident footprint (about 3.2GB).
- `_self_check()` covers the env-supplied-total case. Stays sandbox-runnable.

**Worked example** (why GPU buffered is for the 16GB+ tier):

| Card | Stack: worst-LLM + Kokoro + GPU Parakeet | Ceiling (total−1GB) | STT placement |
|---|---|---|---|
| 12GB Capable | 8912 + 954 + 3200 = **13066** | 11264 | **CPU** |
| 16GB Best | 8912 + 954 + 3200 = **13066** | 15360 | **GPU** |

Both rows use the **worst-case LLM** `max(LLM_PEAK)=8912` (E4B/Better) — the same no-mid-session-thrash
lock `resolve_stt_placement` already uses, so a Fast↔Better swap can never strand placement; only the
card's ceiling differs. A 12GB Capable card therefore stays on CPU STT; a 16GB Best
card can run GPU buffered Parakeet without busting the measured-headroom rule.

### B. `gpu-doctor.sh` — VRAM → profile advice (prints, never writes)
Extends today's nvidia-smi vendor+VRAM detection to print a recommended `.env` snippet, using **named
readonly thresholds** (no magic numbers, matching its current style):

| Detected | Tier | Recommended `.env` snippet |
|---|---|---|
| no GPU / CPU-only | Floor | `STT_ENGINE=buffered` |
| VRAM < 16GB (8/12GB) | Floor→Capable | `STT_ENGINE=buffered`; `STT_FORCE_CPU=1`; `VRAM_TOTAL_MB=<detected>` |
| VRAM ≥ 16GB | Best | `STT_ENGINE=buffered`; `STT_FORCE_CPU=0`; `STT_HEADROOM_MEASURED=1`; `VRAM_TOTAL_MB=<detected>` |

It still **never mutates `.env`, always exits 0** (D16). `resolve_parakeet_device` is the shared
decision logic gpu-doctor uses to *recommend* and the agent's `build_session` uses to *pick the
buffered Parakeet EP* (mirroring how `resolve_stt_placement` picks the STT URL today). Unknown/garbled
VRAM → gpu-doctor recommends the conservative Floor snippet; `placement.py` keeps returning CPU. No
detection bug can route a too-big stack onto a small card.

## Adjacent seams

### `TTS_ENGINE` seam — designed, not built (YAGNI)
Add `TTS_ENGINE` env (default `kokoro`) + a compose profile mirroring `STT_ENGINE`; the agent
generalizes today's `KOKORO_BASE_URL` (`agent/main.py:75`) into a per-engine TTS base-URL lookup.
**Only Kokoro ships.** The v1.3 target is **Chatterbox-Turbo (350M, MIT)** — its streaming path, not the
base 0.5B (offline-only). Avatar caveat (documented decision): non-Kokoro engines have no
`/dev/captioned_speech`, so word-accurate Path-B lip-sync degrades to **Path-A** (the existing
per-utterance fallback); the **voice path is unaffected** (standard OpenAI `/v1/audio/speech`), and the
Phase-14 voice-only invariant holds. Scope guard: **one env + a documented profile**, no `TTSEngine`
base class (no second concrete use today).

### `finalize()` tail-drop investigation — honest scope (not a promised fix)
The canonical NeMo fix regressed in 15a; the two candidates (`STT_THREAD_PRED_OUT`,
`STT_FINALIZE_PAD` in `backend_common.py`) stay **dormant, default-OFF**. R3's deliverable is an
**instrumented GPU runbook + an operator gate (G5)**, not a one-liner. The escape hatch makes this
low-risk: `buffered`/`hybrid` have no lookahead → no held tail → the cut-off is structurally absent in
the new defaults. So R3 either (a) confirms a candidate fixes streaming on GPU under a
**no-WER-regression** gate and promotes it, or (b) documents streaming's tail-drop as *mitigated by the
buffered/hybrid default*. The keep-or-strip-the-dormant-code call is deferred to G5's outcome.

## Docs + headline re-characterization (D15)
R3 updates `CLAUDE.md` / `PROJECT.md` / `README.md` / `STACK.md`: **P50<1.0s is the streaming-mode
guarantee**; hybrid/buffered are accuracy-mode with a measured EOU cost. Adds the **"pick your STT by
hardware"** table (3 modes × 3 tiers) and the honest value framing (D18: cut-off elimination +
robustness, not WER). This reframing is a deliverable, not an afterthought — it is the honest
counterpart to choosing hybrid-default on the GPU tiers.

## Carried risks & measurement gates
1. **Buffered finalize latency on long interview turns is unmeasured** (D12's "measure before locking").
   GPU expected faster; CPU sherpa-onnx Parakeet remains plausible for fallback — exact p50 was not captured. → **G1**.
2. **Hybrid co-residency VRAM** on 12 vs 16GB pins `PARAKEET_GPU_MB` and validates `resolve_parakeet_device`. → **G2**.
3. **int8 WER cost** (~+13% rel) vs the cut-off win is a real-audio judgement, not a benchmark. The
   Floor's *current* streaming-CPU already sits ~7–8% **and** drops words; buffered trades a hair of WER
   for zero word-drop. Confirm on real UAT audio. → **G1**.
4. **Floor CPU contention** — Parakeet-ONNX + LLM (+ optional TTS-CPU) on shared cores; measure P50/feel. → **G1**.
5. **Energy-EOU robustness** — the RMS silence trigger for `buffered` must not fire mid-utterance on a
   pause; tune against `ENDPOINT_SILENCE_MS`. Sandbox-unit-testable on synthetic PCM.

## Per-stream success criteria
- `STT_ENGINE` selects `streaming` / `buffered` / `hybrid`; the supported default is
  `buffered`; `streaming`/`hybrid` remain legacy/manual comparison modes.
- CPU and GPU `buffered` produce **cut-off-free** finals on real UAT audio (G1), within
  the accuracy-mode latency contract.
- `placement.py` derives the VRAM total from env and resolves Parakeet GPU/CPU via the measured fit gate
  (G2); the `STT_HEADROOM_MEASURED` safety lock holds.
- The `TTS_ENGINE` seam exists (Kokoro-only shipped); the `finalize()` tail-drop is fixed-with-no-regress
  or documented-as-mitigated (G5); docs reframe P50<1.0s honestly (D15).

## Plan B/C build refinements (2026-06-28, brainstorm)

Plan A (STT engine seam) is merged. Brainstorming B + C surfaced two simplifications to this design;
both reduce code, neither changes the engine model (D13–D20 stand). **B and C ship as ONE combined
implementation plan** — both are small, they share the docs + VERIFY deliverable, and C's "pick your
STT by hardware" table is the documentation of B's advice logic.

### Delta 1 (Plan B) — hardware awareness is ADVISE-ONLY-IN-BASH (drop `resolve_parakeet_device`)
The buffered-Parakeet device is read **server-side** from `STT_BUFFERED_DEVICE` at container boot
(`backend_parakeet.py`); the agent only chooses which STT *URL* it connects to and cannot change a
running server's onnxruntime EP, and `gpu-doctor.sh` is bash. So the design's "shared
`resolve_parakeet_device(env)` in `placement.py` that `build_session` uses to pick the EP" has **no
runtime consumer** and would duplicate fit-math across Python and bash. Resolution:
- **Dropped:** `resolve_parakeet_device`, the `PARAKEET_GPU_MB` constant, and the agent
  `build_session` change (all from the "Hardware awareness §A" sketch).
- **Kept (smaller):** `placement.py` makes `VRAM_TOTAL_MB` **env-derived** (default 16384) so
  `resolve_stt_placement`'s ceiling is correct on a measured 8/12GB card — consumed by the existing
  GPU-fit gate, so it earns its place. `gpu-doctor.sh` owns the Parakeet-device + engine
  recommendation as **named readonly bash thresholds** (its existing advise-only / always-exit-0 posture).
- **Gate reframe:** **G2** becomes "measure Parakeet GPU footprint to TUNE the gpu-doctor VRAM
  threshold," not "pin the `PARAKEET_GPU_MB` Python constant."

### Delta 2 (Plan C) — DEFER the `TTS_ENGINE` code seam to v1.3 (docs-only now)
The only consumer of a second TTS engine is v1.3 Chatterbox-Turbo (not built; the base 0.5B is
offline-only — the wrong variant). A seam with one implementation violates "no abstraction without
≥2 concrete uses today," and adding the ~10 lines now makes v1.3 no easier. Resolution:
