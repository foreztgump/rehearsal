# Plan 06-01: The working end-to-end Interview Mode loop â€” toggle into Interview, pick a role, get asked one question at a time, answer, get a critique + a strong model answer

## User Story

**As a** learner who wants to rehearse for a security job interview, **I want to** toggle the
trainer into Interview Mode and pick a target role (SOC analyst / security engineer / GRC), **so
that** the agent asks me one realistic, role-relevant question at a time, waits for my spoken
answer, then critiques it and demonstrates a strong model answer before asking the next one.

## Context

This is the **first vertical slice** of Phase 6 â€” the complete demonstrable loop end-to-end
(toggle â†’ role pick â†’ ask Q1 â†’ listen â†’ critique + model answer â†’ next), not a horizontal layer.
After this plan a real user can do the whole interview flow; plan 06-02 then DEEPENS critique
quality (rubric structure) and re-tunes endpointing for slow speech.

**Architecture decision (RESEARCH Â§3, recommendation = Option B):** do NOT build a multi-agent
handoff (`InterviewAgent`/`LearnAgent` swap). Keep the single `HistoryWindowAgent` and steer
behavior by swapping the **system instruction block** between a Learn prompt and an Interview
prompt â€” exactly the persona-hot-swap pattern already proven end-to-end in `handle_persona_update`
(`agent/main.py:308-319`). The mode/role is a **third mutable axis** alongside `current_persona`
that composes into one `update_instructions` call. The toggle is a **one-time, user-initiated
re-prefill** (same cost model as a persona edit) â€” NEVER re-rendered per turn (Pitfall 7).

**State machine = Layer 1 (prompt-only), RESEARCH Â§4 / Â§9 q3:** the Interview system prompt itself
encodes the `ask ONE question â†’ wait â†’ critique â†’ strong model answer â†’ next` contract; the LLM
follows it. Do NOT pre-build the explicit `InterviewState` enum + `next_directive` (Layer 2) â€” only
06-02's strong-vs-weak quality gate decides empirically whether Layer 2 is needed. YAGNI.

**Composition (RESEARCH Â§9 q5):** the render must compose **mode Ă— persona Ă— KB**. When Interview
mode is active the render selects the Interview block (for the picked role) and still appends the
current `session_kb.brief`; when Learn (default) it renders the persona block as today. All three
closures (`handle_persona_update`, `ingest_kb`, `handle_mode_update`) read/write the same mutable
holders so the renders compose â€” never clobber each other.

**Sandbox vs VM split (carried from Phases 1â€“5):** the sandbox CANNOT import livekit / has no
Docker / GPU / Ollama / browser. The **pure** core â€” `agent/interview.py` (`ROLES`,
`render_interview_prompt`, `_self_check`) â€” IS sandbox-verifiable and carries the testable logic.
The `mode.update` RPC round-trip, the ask-Q1 `generate_reply`, and the browser panel are
operator/VM gates (the `[VM-INTROSPECT]` precedent). Marked `autonomous: false` for that reason.

**Scope discipline (YAGNI):** the critique in this slice is whatever the Layer-1 Interview prompt
produces â€” a *basic* "brief critique then a strong model answer" instruction. The **rubric-structured
depth** (technical-accuracy / completeness / precise-terminology / structure assessment) and the
strong-vs-weak quality gate are 06-02. The slow-speech endpointing re-tune is 06-02. Do NOT build
those here; do NOT build the agentâ†’UI state-push attribute (RPC ack only for MVP, RESEARCH Â§9 q4).
