# Plan 04-01: Upload → parse → gate → guard, with a KB-active indicator and clear failure handling

## User Story

**As a** learner starting a session, **I want to** upload PDF/TXT/MD/DOCX files and see them get
picked up (with a doc-count indicator and a clear error if one fails), **so that** the trainer can
later be grounded in my own material — and a bad upload never breaks the conversation.

## Context

This is the **upload + parse + boundary-handling** vertical slice. It delivers the *whole* ingest
path end-to-end (browser file picker → byte stream → agent parse/gate/guard → status back to a
panel) but stops **before** distillation/injection (that is 04-02). With this plan, a user can
upload a file and watch the indicator move `idle → uploading → parsing → ready (n docs)` or show a
clear error — the KB is accumulated in worker memory but not yet injected into the prompt.

**The pure core is sandbox-verifiable.** `agent/kb/parse.py` is a livekit-free module mirroring
`agent/metrics.py`/`agent/persona.py` exactly (frozen module constants, `@dataclass`, typed
returns, a `_self_check()` under `if __name__ == "__main__":`). `parse(name, mime, raw) ->
ParsedDoc | KbParseError` dispatches by type, normalizes encoding, runs the extraction-quality
gate, and runs the token-count size guard — all testable over fixture bytes with no livekit
import. See `04-PATTERNS.md` Pattern A (A1–A4) and `04-RESEARCH.md` §2/§3.3/§4/§5.

**The transport is a `[VM-INTROSPECT]` operator gate.** The byte-stream handler
(`register_byte_stream_handler("kb.upload", ...)`), the `sendFile` client call, and the `kb.state`
participant attribute are grounded in published LiveKit docs + the proven in-repo
attribute-read pattern (`AgentStatePill` reads `lk.agent.state`), but the **exact installed
signatures** cannot be imported in the sandbox. Confirm them on the VM; do NOT mark them passed.
Marked `autonomous: false`.

**Hard rules carried from Phases 1–3** (`04-PATTERNS.md` "Core discipline"):
- **Boundary discipline (CODE_PRINCIPLES §4):** parse is a named boundary — **no bare except, no
  silent swallow**. Every failure returns a **typed `KbParseError`** (never raises across the
  boundary); the agent sets a `kb.state` error and the voice loop keeps running with the
  **unchanged** prefix (the `KB_SLOT` stays empty — injection is 04-02).
- **Local-first / no egress (PERF-03):** the file rides the existing LiveKit room data plane as a
  byte stream — **no new HTTP upload endpoint**, nothing leaves the LAN.
- **Ephemeral (KB-06):** parsed docs live **only** in a small in-memory worker object; if a temp
  file is written during parse, it is deleted in a `finally`. No disk/db persistence.
- **Pinned deps, no `:latest`:** `pymupdf4llm`, `pymupdf`, `python-docx` pinned tight.
- **DOCX via `python-docx` only** — pymupdf4llm's Office support needs paid PyMuPDF Pro
  (`04-RESEARCH.md` §3.2). PDF/TXT/MD are free via pymupdf4llm/stdlib.
- **`agent/metrics.py` is untouched** (per-turn JSON key contract frozen).
