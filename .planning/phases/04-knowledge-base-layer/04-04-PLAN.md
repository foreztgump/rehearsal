# Plan 04-04: Close the two HIGH UAT gaps — effective context = 8192 (not 4096) and a distill brief that actually carries the FACTS: anchor the trainer cites

## User Story

**As a** learner who uploaded material, **I want** the trainer to actually reference the exact
facts in my docs (codenames, audit IDs) AND for large KBs to fit the context window without silent
truncation, **so that** the inline-and-cache KB design (KB-04 grounding + KB-05 flat-TTFT) works
end-to-end on the live RTX 5090 stack — not just in sandbox accounting.

## Context

Phase-4 code was executed, committed, and sandbox-verified; the keystone operator runbook
(`04-KB-VERIFY.md`) was authored. **Conversational UAT then ran scriptable proxies against the LIVE
local Docker stack** (RTX 5090, Ollama `gemma3:4b-it-qat`, rebuilt agent image) and surfaced **2
HIGH gaps** (`04-UAT.md` Gaps section) plus one MINOR harness bug. 4 tests passed, 1
pass-with-caveat, 1 voice-gated/blocked. This plan closes the two gaps with **minimal, targeted**
edits (MVP_MODE=true — no speculative abstractions) and re-runs the SAME UAT proxies against the
rebuilt stack as the verification step.

**GAP-1 [HIGH] — effective Ollama context is 4096, half the pinned `num_ctx` 8192.** The live
runner loaded `--ctx-size 8192 --parallel 2` → **4096 tokens per slot**; real KB distill prompts
logged `truncating input prompt limit=4096 prompt=6077 keep=4 new=4096` (×2) — the frozen prefix
the `num_ctx` pin was meant to protect is silently truncated. Two coupled causes:
- *Cause 1:* the `ollama` service in `docker-compose.yml` pins neither parallelism nor context —
  `OLLAMA_NUM_PARALLEL` is unset (defaults to **2**) and `OLLAMA_CONTEXT_LENGTH=4096` is in effect.
- *Cause 2:* the Modelfile `PARAMETER num_ctx 8192` never applies on the hot path — the agent loads
  the **base tag** `gemma3:4b-it-qat` via `openai.LLM.with_ollama(model=resolved_llm_tag())` and
  `distill()` POSTs the base tag to `/api/generate`, **NOT** the `adept-gemma` pinned model, so the
  Modelfile PARAMETER is moot end-to-end.

**GAP-1 chosen approach — service-env (single source of truth), NOT repointing to `adept-gemma`.**
Set `OLLAMA_NUM_PARALLEL=1` and `OLLAMA_CONTEXT_LENGTH=8192` on the `ollama` service.
**Rationale:**
1. *Parallelism must be pinned in the service env regardless of approach.* Effective per-slot
   context = `ctx-size / num_parallel`. Even if the hot path were repointed to `adept-gemma` (whose
   Modelfile pins `num_ctx 8192`), with `num_parallel=2` the per-slot context is still 4096 and the
   truncation persists. So the `adept-gemma` route does **not** fix the gap without *also* touching
   the service env — making the Modelfile pin redundant for enforcement.
2. *One source of truth covers BOTH call paths.* `OLLAMA_CONTEXT_LENGTH` governs the default context
   for every model loaded over `/v1` (the agent hot path) **and** `/api/generate` (the off-hot-path
   distill), independent of tag. Repointing only `resolved_llm_tag()`→`adept-gemma` would fix the
   `/v1` path but the env still governs both; the env is the simpler, tag-agnostic lever.
3. *No extra deploy step / no stale-deploy risk.* Approach (a) requires `ollama create adept-gemma`
   at every deploy and an `OLLAMA_MODEL` repoint; the stale-deploy guard (runbook §0) already bit
   once this phase. The env change ships in `docker-compose.yml` with the existing `up -d`.
4. *VRAM-neutral coupling.* `--ctx-size 8192 --parallel 2` already pre-allocates **8192 tokens of
   KV total** (2 slots × 4096). `--ctx-size 8192 --parallel 1` pre-allocates **8192 tokens of KV
   total** (1 slot × 8192) — the same total KV footprint, just usable as one contiguous window. So
   raising the *effective* per-request context from 4096→8192 is expected to be VRAM-neutral. This
   MUST still be confirmed: Proof D peak was **10196 MB at 4096 effective**; re-run
   `./scripts/vram-validate.sh --with-kb` and a real-KB `nvidia-smi` to confirm peak stays
   **< 15360 MB** with q8_0 engaged. The Modelfile `num_ctx 8192` is RETAINED (it documents the
   budget and governs `adept-gemma` if ever created for the VRAM validator), with a comment noting
   the hot path is enforced by the service env.

**GAP-2 [HIGH] — distill output ignores the FACTS-anchor contract; KB-04 grounding not
demonstrated.** `distill()` on `gemma3:4b-it-qat` returned **meta-commentary** ("This is a good,
comprehensive specification… Strengths: …") instead of the mandated `DOMAIN BRIEF` + `FACTS:`
verbatim-anchor line that `DISTILL_INSTRUCTION` asks for. The codename survives verbatim in the
prose, but there is **no `FACTS:` line**, and when asked the trainer **deflects** (Socratic) rather
than citing `BLUE-OTTER-7714` / `AUD-9931-X`. KB-04 fact-anchoring (Pitfall 8) depends on that
`FACTS:` line. The fix has three minimal levers, applied together:
1. **Tighten/few-shot `DISTILL_INSTRUCTION`** (distill.py) — forbid evaluation/critique
   ("do not say whether the material is good"), demand the two sections, and show a tiny worked
   example so the small instruct model emits the `FACTS:` line.
2. **Post-validate + repair** (distill.py) — after the call, assert the brief contains a line
   starting with `FACTS:`; if absent, run **one** focused off-hot-path repair call that extracts a
   `FACTS:` line from the source and append it; if STILL absent, raise `DistillError` (the existing
   typed boundary — session continues without KB, REL-03). Off-hot-path, so the extra call is
   invisible to the voice loop.
3. **Persona cite-nudge** (persona.py) — when (and ONLY when) a non-empty brief is present, prepend
   a fixed instruction to cite the supplied material's exact terms when relevant. Must keep
   `render_prompt(p, "") == EXPECTED_DEFAULT` byte-identical (the golden seam) and stay byte-stable
   across turns (Pitfall 7) — the nudge is a frozen constant, deterministic, no volatile data.

**MINOR (not a gap)** — `ollama/warmup.py` defaults `WHISPER_MODEL` to
`Systran/faster-whisper-large-v3-turbo`, but `agent/main.py` pins `Systran/faster-whisper-large-v3`;
the turbo model isn't in the offline HF cache, so `./scripts/vram-validate.sh --with-kb` 500s on
whisper warmup until `WHISPER_MODEL` is overridden. Align the warmup default to `large-v3`.

**Constraints carried into every task:** `agent/metrics.py` is a FROZEN read-only contract — do NOT
touch it. Respect the coupled constants `BRIEF_TOKEN_BUDGET` (distill.py),
`KB_MAX_TOKENS`/`KB_WARN_TOKENS` (parse.py), `num_ctx` (Modelfile) — changing one without the others
is flagged as a bug in the code comments. After fixes the stack MUST be rebuilt
(`docker compose build agent web && docker compose up -d`) before re-verification — the stale-deploy
guard (runbook §0) already bit once this phase. `autonomous: false` — the live re-verification
proofs are operator/VM gates.
