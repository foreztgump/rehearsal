# Plan 04-03: Prove the flat-TTFT invariant holds with a KB loaded, and pin num_ctx under the 16GB floor

## User Story

**As the** operator validating Adept, **I want to** prove that loading a large KB costs prefill
**only on turn 1** (turn-2 TTFT ≪ turn-1, and ≈ no-KB turn-2) and that three models + the larger
KV cache still co-reside under 16GB, **so that** the inline-and-cache design is verified, not
assumed — the keystone invariant of the whole phase.

## Context

This is the **keystone verification slice** and is **operator/VM-only** — the sandbox has no
Docker/GPU/Ollama/browser (carried discipline from Phase 1's VRAM gate and Phases 2/3's
`[VM-INTROSPECT]` deferrals). It owns **Success Criterion 3 / KB-05**: per-turn TTFT stays flat
whether or not a KB is loaded, because the brief is injected once and held in Ollama's prefix/KV
cache. There is almost no new code — the proof READS the existing `agent/metrics.py` `llm_ttft_ms`
(per-turn, keyed by `speech_id`) and cross-checks Ollama prompt-eval logs; the only edits are the
`num_ctx` pin (`ollama/Modelfile`), a KB-loaded re-run path for the existing
`scripts/vram-validate.sh`, and a new operator-runbook doc capturing the exact commands + measured
results.

**Three things to establish** (`04-PATTERNS.md` Patterns F + I; `04-RESEARCH.md` §3.5/§9/§10.3):

1. **Flat-TTFT proof (KB-05):** load a LARGE KB, then compare **turn-1 `llm_ttft_ms`** (cold
   prefill of the brief — the one sanctioned re-prefill, expected `over_budget: ["llm_ttft"]`) vs
   **turn-2+ `llm_ttft_ms`** (`≪`, cached). Also compare KB-loaded turn-2 vs no-KB turn-2 (should
   be ~flat). Cross-check Ollama logs: turn-2 shows a SMALL prompt-eval "new tokens" count, not a
   full re-eval of the brief (a full re-eval = Pitfall 7 cache-bust warning sign).
2. **`num_ctx` pin (Pattern F):** Ollama PRE-ALLOCATES the full `num_ctx` KV cache upfront, so the
   value is a direct VRAM cost. Size it to the measured worst case `≈ persona(~250) + brief budget
   (~1500) + max history window + headroom`. Keep `8192` if the measured worst case fits; bump only
   to a measured value. `BRIEF_TOKEN_BUDGET` (distill.py), `KB_MAX_TOKENS`/`KB_WARN_TOKENS`
   (parse.py), and `num_ctx` are **coupled constants — set them together** after measuring the real
   distilled-brief token count on the VM.
3. **KB-load VRAM re-check (PERF-02 re-validation):** re-run `scripts/vram-validate.sh` **with a KB
   loaded** — KB load is the peak-memory moment. Confirm peak used-VRAM < 16384 MB (with the 1GB
   headroom the script already enforces), `q8_0` KV still engaged (FAIL LOUDLY if it fell back to
   F16), and exactly 3 GPU procs (no embedder/vector store).

**`agent/metrics.py` is NOT edited** — the frozen per-turn key set
(`eou_ms/stt_ms/llm_ttft_ms/tts_ttfb_ms/e2e_ms/over_budget`) is the Phase-3 contract; 04-03 only
READS it. Marked `autonomous: false` — every acceptance below that touches the live stack is an
explicit OPERATOR-VERIFICATION step, NOT an auto-passing sandbox check.
