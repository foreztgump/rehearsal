# Plan 01-03: Self-hosted LiveKit + turn-detector + per-stage metrics scaffold

## User Story

**As a** privacy-strict operator, **I want to** run LiveKit fully self-hosted with correct LAN
ICE config and a local turn-detector whose weights are baked into the agent image, plus a
per-stage metrics scaffold that emits one real timing, **so that** no call ever leaves the LAN
and the flat-TTFT invariant has measurement infrastructure from turn one — before any voice
flows.

## Context

GREENFIELD. Follow templates **T3** (LiveKit self-host config), **T4** (local turn-detector +
weights bake), **T5** (AgentSession wiring), **T6** (per-stage metrics scaffold) from
`01-PATTERNS.md`. Depends on 01-01 (compose/proxy/TLS) and 01-02 (resolved OLLAMA_MODEL +
warmup TTFT). This plan closes the walking skeleton: LiveKit server config with LAN-correct
ICE, a custom agent image that bakes VAD + turn-detector weights offline, an `AgentSession`
constructed against the three local model endpoints, and the metrics scaffold that routes the
01-02 warmup LLM TTFT as the **"one real metric emitted"** gate. No live voice turn yet.

Key risks from RESEARCH.md handled here:
- `inference.TurnDetector` defaults to **LiveKit Cloud** → must use local `MultilingualModel`.
- Default LiveKit advertises the container/internal IP → media never flows; fix with
  `use_external_ip` / `node_ip` + open UDP (prefer udp mux 7882).
- Session-level `metrics_collected` is **DEPRECATED** → subscribe per-plugin.
- Turn-detector import path on `~=1.5` must be verified (`.multilingual` vs string).
