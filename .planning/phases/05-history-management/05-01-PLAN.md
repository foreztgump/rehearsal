# Plan 05-01: Cap the conversation history ITEM list each turn so per-turn TTFT stays flat — without ever touching the frozen persona+KB prefix

## User Story

**As a** learner holding a long spoken session with the trainer, **I want** the conversation
history to be sliding-windowed as it grows, **so that** per-turn time-to-first-token stays flat
turn-after-turn (it never climbs with turn count) and the heavy KB/persona prefix cache keeps
hitting.

## Context

This is the **single vertical slice** for Phase 5 (the roadmap defines exactly one plan, 05-01).
It owns **SESS-05** and the three success criteria: (1) history is sliding-windowed as the session
grows, (2) per-turn TTFT stays flat across a long session, (3) history management sits **behind**
the frozen persona/KB prefix and does not invalidate the prefix cache.

**The keystone discipline (carried from Phases 3–4, do not break):** the frozen `[persona] +
[KB brief]` block lives in the Agent's `instructions` (`render_prompt` output, `agent/main.py:262`).
Conversation turns are separate `ChatContext` **items** LiveKit appends AFTER the instructions.
Phase 5 mutates **only the item list** and **NEVER calls `update_instructions`** — that is what makes
"behind the frozen prefix" (Criterion 3) structurally guaranteed: `ChatContext.truncate()` "always
preserves system instructions" (LiveKit docs), so windowing the items is *physically incapable* of
touching the cached persona+KB prefix. The two sanctioned `update_instructions` re-prefills
(`handle_persona_update` at `main.py:287`, `ingest_kb` at `main.py:362`) are untouched; Phase 5 adds
none.

**Why this keeps TTFT flat (Pitfall 10 — the thing this phase fixes):** append-only history creeps
(`llm_ttft_ms` climbs turn-over-turn) and cliffs at `num_ctx` (a session wrapping the window evicts
the ENTIRE KB/persona prefix and re-prefills it). Capping history so `persona(~250) + KB
brief(≤1500) + history + turn` stays well under `OLLAMA_CONTEXT_LENGTH=8192` (the Modelfile already
reserves ~5000 tok for history) bounds the per-turn prefill. Cut from the **FRONT** (drop oldest
items) — the cache-safe edge; never rewrite the middle (maximizes re-prefill).

**Sandbox vs VM split (carried from Phases 1–4):** the sandbox CANNOT import livekit / has no
Docker / GPU / Ollama / browser. The **pure** core — `agent/history.py` windowing decision logic
(`_self_check`) and the unchanged `render_prompt` byte-stability golden — IS sandbox-verifiable and
carries the client-verifiable acceptance. The LiveKit hook round-trip and the **flat-TTFT-over-time
proof** are operator/VM gates (deferred like 04-03 / the `[VM-INTROSPECT]` pattern), captured in
`05-HISTORY-VERIFY.md`. Marked `autonomous: false` for that reason.

**Scope discipline (YAGNI / CODE_PRINCIPLES §7):** async summarization is the OPTIONAL
behind-the-window stretch; window-only satisfies SESS-05's "sliding-window **and/or** summarized" and
is fully deterministic. **Summarization is OUT of this MVP slice** — do not build it. It is noted as
a future refinement only. An optional token-budget gate (defensive against an unusually long single
turn) is likewise a documented refinement, NOT built here — the item-count window is the floor.
