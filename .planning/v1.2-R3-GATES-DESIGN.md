# v1.2 — R3 — Operator-Gate Enablement (make G1–G5 runnable) — Design

## Summary

R3's five operator gates (`v1.2-R3-VERIFY.md`) cannot be run today: they are blocked on **missing code**,
not missing measurements. The buffered/hybrid engine body (`backend_parakeet.load_model`/`finalize`)
is `NotImplementedError`, the int8 weights are not baked (only `# GATE:` comments in both Dockerfiles),
there is no WER tooling, and no committed audio. This stream **closes every gap that does not need a
GPU**, in the sandbox, validated by runnable checks — so the RTX 5090 session becomes purely
*build images → run → record numbers*. It mirrors the Phase 9/10/R2 "code-complete → operator-gate"
pattern this project already uses.

**Scope decision (D21):** build everything buildable now; do not hand the operator a runbook that
still contains "implement the ONNX decode here."

## Decisions (continue the R3 sequence D13–D20)

| # | Decision | Rationale |
|---|---|---|
| D21 | Build all gate-enabling code in the **sandbox now** (code-complete → operator-gate), not a runbook-only handoff | The hard part (the decode body + a reproducible WER harness) is buildable without a GPU; doing it in the sandbox makes it testable, reviewable, version-controlled, and reused on every rebuild. Matches Phases 9/10/R2. |
| D22 | `backend_parakeet` uses the **sherpa-onnx `OfflineRecognizer`** (k2-fsa), not a hand-rolled ORT+numpy TDT decode. **Supersedes D17.** | sherpa-onnx is the canonical Parakeet-TDT-ONNX runtime; it owns mel + encoder + TDT greedy decode, so the backend shrinks to ~15 lines of config (no blind decode loop to write/validate). Its **CPU EP runs a real transcribe in the sandbox** → real int8 WER now, not stubs. Cost: a `sherpa-onnx` wheel in the STT images + a STACK pin. |
| D23 | Commit a **small CC-BY LibriSpeech test-clean fixture set + ground-truth transcripts** for reproducible WER; the operator records the **cut-off clip at gate time** (own voice, not committed — privacy) | Makes the WER number reproducible run-to-run and runnable in the sandbox (sherpa-onnx CPU). The streaming-WER baseline is then measurable on the SAME clips on the GPU. Keeps the operator's voice out of git. |
| D24 | **G2's 12GB threshold is derived arithmetically** from the 5090-measured Parakeet GPU footprint; the 12GB boundary is not physically tested (no 12GB card available) | Only a 24GB RTX 5090 exists. Measuring Parakeet's actual GPU MB + plugging into the existing fit math conservatively sets `VRAM_HYBRID_MB`; a physical 12GB run is unnecessary for a conservative threshold. Documented limit. |

> **Ponytail cuts** (deliberate simplifications): (a) **no latency-replay driver** — reuse the existing
> `dur_ms` + `metrics.py` rolling summary + live-talk (Phase-14 precedent); add a driver only if log-reading
> proves insufficient. (b) **cut-off check folded into `scripts/stt-wer.py`** as a `--cut-off` mode, not a
> separate script. (c) **one shared `fetch_parakeet_onnx.sh`** reused by both Dockerfiles + the sandbox
> (earns its place: ≥2 callers).

## Architecture — what gets built now (sandbox)

### 1. Parakeet engine — closes G1's code gap  (`stt/backend_parakeet.py`)
Replace the two `NotImplementedError` bodies with a thin sherpa-onnx wrapper; keep everything else
(`STREAMS=False`, `decode_chunk` no-op, server-owned `_turn_pcm`, KeyError→SystemExit posture).

- `load_model()` → `sherpa_onnx.OfflineRecognizer.from_transducer(encoder=<dir>/encoder.int8.onnx,
  decoder=<dir>/decoder.int8.onnx, joiner=<dir>/joiner.int8.onnx, tokens=<dir>/tokens.txt,
  num_threads=<env>, sample_rate=16000, feature_dim=80, decoding_method="greedy_search",
  model_type="nemo_transducer", provider=("cuda" if STT_BUFFERED_DEVICE=="gpu" else "cpu"))`.
  `<dir>` = `STT_PARAKEET_MODEL` (already wired). Empty dir → `SystemExit` (unchanged posture).
- `finalize(model, state)` → int16 `_turn_pcm` → float32 numpy in [-1, 1] → `stream =
  model.create_stream(); stream.accept_waveform(16000, samples); model.decode_stream(stream);
  return stream.result.text` (native punct/caps as-is). Empty buffer → `""` (unchanged).
- **Heavy `import sherpa_onnx` / `import numpy` stays INSIDE the callables** so the module still
  byte-imports in a sherpa-less sandbox — preserves `test_engine.py`'s "imports without ORT" guarantee.

> **Implementer note:** pin the `sherpa-onnx` version, then confirm the exact `from_transducer` keyword
> names + `accept_waveform` sample format against that version (the kwargs above match the documented
> sherpa-onnx Python API and the C-API config dump; verify on pin). The expected sample format is float32
> normalized to [-1, 1] (`normalize_samples=True` in the model's feature config).

### 2. Weights + bake  (`scripts/fetch_parakeet_onnx.sh`, both Dockerfiles)
- **`scripts/fetch_parakeet_onnx.sh`** — download the **pinned** prebuilt bundle
  `sherpa-onnx-nemo-parakeet-tdt-0.6b-v2-int8` (GitHub release tarball or HF mirror) into a target dir,
  extract, and assert the 4 files exist (`encoder.int8.onnx`, `decoder.int8.onnx`, `joiner.int8.onnx`,
  `tokens.txt`). Build-time download, run offline thereafter (mirrors `export_onnx.py`'s
  snapshot-then-offline pattern). Reused by both Dockerfiles + the sandbox harness.
- **`stt/Dockerfile.cpu`** — replace lines 70–76 `# GATE:` comment with: `pip install sherpa-onnx`
  (CPU wheel) + `RUN scripts/fetch_parakeet_onnx.sh /models/parakeet-tdt-0.6b-v2-onnx-int8`.
- **`stt/Dockerfile`** (GPU) — replace its `# GATE:` comment with the CUDA-enabled `sherpa-onnx` install
  + the same fetch. (Blackwell sm_120 EP is an operator risk — see Risks.)

### 3. WER + cut-off harness — closes G1/G5 measurement gap  (`scripts/stt-wer.py`, `tests/fixtures/asr/`)
- **`tests/fixtures/asr/`** — ~3–6 CC-BY LibriSpeech test-clean utterances (16 kHz mono WAV, a few
  seconds each) + `transcripts.jsonl` (`{"clip": "...wav", "text": "<ground truth>"}`) + a `README`/LICENSE
  note with LibriSpeech CC-BY-4.0 attribution. ~1–3 MB total.
- **`scripts/stt-wer.py`** — runs Parakeet (sherpa-onnx CPU) over the fixtures, computes WER via
  `jiwer` (dev dep, NOT in the image), prints per-clip + aggregate. Modes:
  - default → aggregate int8-Parakeet WER on the fixture set (**real number, sandbox CPU**).
  - `--cut-off <clip> <ref>` → objective trailing-word diff (last-N hypothesis vs reference tail) to flag
    end-of-speech word-drop — used on the operator's recorded 15a clip at gate time.
  - `--self-check` → run one fixture clip, assert WER below a loose ceiling (e.g. < 15 %). This **is** the
    ponytail runnable check; no separate `test_*.py` needed.
  - On the GPU host the operator runs the same harness against the **streaming** finals on the same clips
    to fill G1's "vs streaming" column.

### 4. Hybrid VRAM — closes G2  (`scripts/vram-validate.sh`)
- Add the `STT_ENGINE=hybrid STT_BUFFERED_DEVICE=gpu` case to the existing script: warm
  Nemotron + Kokoro + worst-LLM + Parakeet-on-CUDA, sample `nvidia-smi memory.used` at the KB-prefill
  peak, print the measured peak **and the derived Parakeet footprint** (`hybrid_peak − streaming_peak`),
  then print the recommended `VRAM_HYBRID_MB`. The 12 GB boundary is **arithmetic from the measured
  footprint** (D24) — keep the script's existing assertion + named-threshold style.

### 5. Rewrite `v1.2-R3-VERIFY.md` into a deterministic runbook
Each gate becomes "run **THIS** script → read **THIS** number → paste into **THIS** field," every step
tagged `[sandbox]` or `[gpu]`. Update prereqs (sherpa-onnx dep, `fetch_parakeet_onnx.sh`, fixtures).

### 6. Docs
- `v1.2-R3-DESIGN.md` — annotate D17 as revised by D22 (sherpa-onnx).
- `.planning/research/STACK.md` — pin `sherpa-onnx` version, the prebuilt Parakeet ONNX bundle
  repo/revision, and `jiwer` (dev). Never `:latest`.
- `CLAUDE.md` / `README.md` only if a convention changes (new dev dep group; fixtures dir).

## Sandbox vs GPU split (the payoff)

| Buildable + checkable in sandbox NOW | Irreducibly GPU-only (operator, mechanical) |
|---|---|
| `backend_parakeet` sherpa-onnx body (real CPU EP) | G1: **GPU-ONNX** finalize p50 (CUDA EP) on long turns under contention |
| Real int8-Parakeet WER on committed fixtures (`--self-check`) | G1: **streaming-WER** on the same fixtures (needs NeMo/GPU) |
| Cut-off harness (`--cut-off`) | G2: actual VRAM peak + Parakeet GPU footprint on the 5090 |
| `fetch_parakeet_onnx.sh` + Dockerfile bake wiring | G3: hybrid EOU→final p50 live (read rolling-summary logs) |
| `vram-validate.sh` hybrid extension (script) | G4: streaming P50 on real audio + frames-identical confirm |
| `test_dispatch.py` / `test_engine.py` (already pass) | G5: GPU A/B the two dormant finalize candidates on the 15a clip |
| Rewritten VERIFY runbook + doc/STACK updates | Blackwell sm_120 EP sanity for sherpa-onnx CUDA |

## Per-gate enablement

- **G1** — *was:* implement engine + measure. *Now:* engine implemented (sherpa-onnx) + weights baked +
  WER harness + fixtures all in-repo; sandbox yields the int8 WER + cut-off result; operator records
  GPU-ONNX finalize p50 + streaming-WER on the 5090.
- **G2** — `vram-validate.sh` hybrid case prints the measured Parakeet footprint; operator pastes it and
  the derived `VRAM_HYBRID_MB` (D24).
- **G3** — no new code; the server-stamped `dur_ms` + `metrics.py` rolling p50/p95 already capture
  EOU→final. Runbook gives the `docker compose logs agent | grep rolling_summary` recipe.
- **G4** — already sandbox-green (`test_dispatch.py` + `test_engine.py`); operator adds the streaming-P50
  read on real audio.
- **G5** — no new code (candidates dormant in `backend_common.py`); the `--cut-off` harness scores the
  GPU A/B objectively. Buffered/hybrid being the GPU-tier defaults (no lookahead) makes "mitigated" the
  likely close.

## Testing / runnable checks (ponytail: one check per non-trivial piece)
- `python3 stt/test_engine.py` — extend the stub Parakeet backend so the buffered/hybrid composition
  asserts the sherpa-onnx wrapper contract **without** sherpa installed (module still byte-imports).
- `python3 scripts/stt-wer.py --self-check` — real Parakeet transcribe over one fixture clip on CPU,
  assert WER < loose ceiling. Proves the engine end-to-end in the sandbox (requires sherpa-onnx CPU +
  fetched weights here; if sandbox network blocks either, this check runs on the GPU host instead and the
  sandbox falls back to stub-only — noted, not silently skipped).
- `python3 stt/test_dispatch.py` — unchanged; streaming dispatch byte-identical (G4 sandbox half).
- `scripts/vram-validate.sh` keeps its assertion style; the new hybrid case runs on the GPU.

## Risks & honest limits
1. **No 12 GB card** → G2 derives the threshold, can't physically test the 12 GB boundary (D24; conservative fit math — acceptable).
2. **sherpa-onnx CUDA on Blackwell sm_120** is unproven — same risk class as the existing
   onnxruntime-gpu-on-5090 gate. CPU EP is safe; hybrid can fall back to `STT_BUFFERED_DEVICE=cpu`
   (already supported), so the gate still closes even if the CUDA EP misbehaves.
3. **Sandbox network** — fetching the sherpa-onnx wheel + weights at build/setup needs network. Committing
   the audio fixtures removes the audio half; if the wheel/weights can't be fetched here, real-WER-in-sandbox
   degrades to stub tests + GPU-only WER (the harness + fixtures are committed regardless).
4. **WER "vs streaming"** — streaming WER needs NeMo/GPU; the sandbox produces the Parakeet-int8 number vs
   ground-truth, the operator measures streaming on the same clips on the 5090.

## Non-goals (unchanged from R3)
- No `TTS_ENGINE` code, no `resolve_parakeet_device`, no `PARAKEET_GPU_MB` Python constant.
- No NeMo-fp16 Parakeet, no multilingual (v3), no live in-app engine switching.
- No change to the `streaming` default behaviour (G4 guards byte-identity).

## Per-stream success criteria
- `backend_parakeet` runs a **real** Parakeet transcribe (sherpa-onnx) on CPU in the sandbox; the
  buffered/hybrid composition tests pass without sherpa installed.
- `scripts/stt-wer.py --self-check` produces a real int8 WER on the committed fixtures (or is documented
  as GPU-deferred if sandbox network blocks the fetch).
- Both Dockerfiles bake the pinned int8 bundle + sherpa-onnx (no `# GATE:` placeholder remains).
- `vram-validate.sh` has a hybrid-GPU case that prints the measured Parakeet footprint + recommended
  `VRAM_HYBRID_MB`.
- `v1.2-R3-VERIFY.md` is a deterministic, step-tagged runbook; STACK.md pins are in place; D17 is annotated
  as revised by D22.
